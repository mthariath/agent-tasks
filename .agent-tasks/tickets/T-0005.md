## Context

`agent-tasks` should cooperate with common agent conventions such as `AGENTS.md`, `PLANS.md`, and `DESIGN.md` without trying to replace them. Tickets and epics need a clean way to point at supporting docs, and the reference TUI should surface those docs without owning their schemas.

## Outcome

Tickets and epics can reference supporting repo files, the reference tool validates and surfaces those references, and the default workflow guidance teaches agents to read them when relevant.

## Acceptance Criteria

- Tickets and epics support simple repo-relative file references.
- Validation catches missing, absolute, or out-of-repo references.
- CLI and MCP surface references in entity payloads and allow setting them during creation.
- Ticket and epic detail views render a references section when present.
- Default managed guidance and workflow skills tell agents to read referenced `PLANS.md` / `DESIGN.md` style docs when relevant.
- The standard/docs make the ownership boundary clear:
  - `.agent-tasks/` owns workflow/task state
  - `AGENTS.md` provides passive repo-wide guidance
  - `PLANS.md` and `DESIGN.md` remain external conventions

## Notes

Keep the reference model generic:
- no typed reference taxonomy yet
- plain repo-relative paths only
- no deep parsing of foreign file schemas

Implemented:
- added top-level `references` support for tickets and epics in the core model, create/edit paths, and validation
- validate repo-relative references eagerly on mutation and during indexing
- surfaced references in CLI `show`, create/edit flags, MCP entity payloads, and MCP create/update tools
- added a dedicated references section to ticket and epic detail in the TUI
- updated the default workflow guide, managed `AGENTS.md` block, and installed/source workflow skills to tell agents to read relevant supporting docs
- documented the interop boundary for `AGENTS.md`, `PLANS.md`, and `DESIGN.md`
