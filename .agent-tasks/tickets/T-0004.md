## Context

`agent-tasks` can already plan work in parallel, but it does not yet define or enforce how multiple agents should claim tickets, create isolated workspaces, and finish work safely on one machine.

## Outcome

The reference tool has a first local execution model with exclusive claims, worktree-backed start flows, runtime coordination state, lifecycle commands, and matching MCP tools.

## Acceptance Criteria

- `project.yaml` can define review semantics needed for finish/handoff flows.
- Runtime coordination state lives under `.agent-tasks/.runtime/` and is ignored by git.
- The core layer can claim, start, block, finish, release, and inspect execution state for a ticket.
- Starting coding work defaults to a git worktree with a predictable branch and path.
- CLI commands exist for `claim`, `start`, `block`, `finish`, `release`, `serve`, and execution validation.
- MCP exposes matching execution tools.
- Execution state is validated and surfaced in the reference TUI at least in ticket detail.
- The behavior is documented in an execution-model doc and covered by core/CLI tests.

## Notes

Defaults chosen for this pass:
- single-machine only
- exclusive claim per coding ticket
- worktrees by default for actual coding work
- finish means ready for integration/review, not necessarily merged

Implemented:
- added runtime coordination indexing and lifecycle operations in core for `claim`, `start`, `block`, `finish`, `release`, and execution validation
- added local runtime files under `.agent-tasks/.runtime/` plus safe init-managed `.gitignore` integration
- added CLI execution commands and a local `serve` coordinator
- added MCP execution tools for claim/start/block/finish/release plus execution status queries
- documented the model in `docs/EXECUTION_MODEL.md` and updated CLI/spec/readme docs
- added core and CLI tests that exercise real git repos, worktree starts, finish/handoff flows, MCP tools, and runtime validation
- surfaced execution state in ticket detail in the TUI
